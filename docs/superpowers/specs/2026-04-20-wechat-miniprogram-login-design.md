# WeChat Mini Program Login Design

## Summary

Implement WeChat login for IceZone Studio web app via a personal WeChat Mini Program acting as a "login bridge." Users scan a QR code on the web login page with WeChat, confirm in the Mini Program, and get logged into the web app automatically via Supabase Realtime.

## Constraints

- Individual developer (personal entity Mini Program)
- No enterprise certification — cannot use WeChat web OAuth directly
- Only need openid (no nickname/avatar from WeChat)
- WeChat accounts are independent from existing Google/email accounts
- Free tier Supabase

## Architecture

```
┌─────────────┐         ┌──────────────────┐        ┌─────────────────┐
│  Web Frontend│         │  Supabase        │        │  WeChat Server  │
│  (Next.js)  │         │                  │        │                 │
│             │         │  Edge Function   │◄──────►│  code2session   │
│  QR Code    │         │  (wechat-login)  │        │  API            │
│  Realtime   │◄───────►│                  │        └─────────────────┘
│  Subscribe  │         │  wechat_login_   │
│             │         │  sessions table  │
└─────────────┘         │  auth.users      │
                        └──────────────────┘
       ▲                        ▲
       │                        │
       │    ┌──────────────┐    │
       │    │  WeChat Mini  │    │
       └────│  Program     │────┘
            │  wx.login()  │
            └──────────────┘
```

## Core Flow

1. Web generates UUID, displays QR code (content: Mini Program path with UUID param)
2. User scans QR with WeChat → opens Mini Program
3. Mini Program calls `wx.login()` → gets code
4. Mini Program POSTs `{code, uuid}` to Edge Function
5. Edge Function calls WeChat `code2session` API → gets openid
6. Edge Function finds/creates Supabase user → generates session tokens
7. Edge Function updates `wechat_login_sessions` row: status='confirmed' + tokens
8. Web receives Realtime update → `setSession()` → redirect to dashboard

## Database Design

### `wechat_login_sessions` table

| Field | Type | Description |
|-------|------|-------------|
| id | uuid (PK) | The UUID generated by web frontend |
| status | text | `pending` → `confirmed` / `expired` / `failed` |
| user_id | uuid (FK → auth.users) | Set on successful login |
| access_token | text | Supabase session access token |
| refresh_token | text | Supabase session refresh token |
| error_message | text | Error description on failure |
| created_at | timestamptz | Creation time |
| expires_at | timestamptz | Expiry time (created_at + 5 minutes) |

### State Machine

```
pending ──► confirmed   (successful login)
   │
   ├──► expired        (5 min timeout, no scan)
   │
   └──► failed         (WeChat API error / openid fetch failure)
```

### RLS Policy

- Anonymous users can INSERT (create pending row) and SELECT (subscribe to own UUID)
- No client-side UPDATE allowed
- Edge Function uses service_role key to update status

### Cleanup

- pg_cron job runs hourly: DELETE rows where `created_at < now() - interval '1 hour'`

## Timeout & Failure Handling

### Web Frontend
- Sets `expires_at = now() + 5min` on session creation
- Local 5-minute countdown timer alongside Realtime subscription
- Timer expires + still pending → show "QR expired, click to refresh"
- User clicks refresh → new UUID, discard old record

### Edge Function
- Validates `expires_at > now()` before calling WeChat API
- WeChat API failure → update status='failed', write error_message
- User creation / session generation failure → same treatment

### Mini Program
- 10s request timeout → show "Login failed, please retry"
- Error response → show specific error + retry button

## Security

### AppSecret Protection
- Stored only in Edge Function env vars (`WECHAT_APP_ID`, `WECHAT_APP_SECRET`)
- Never in frontend code or git

### Replay Prevention
- Each UUID consumed once: `UPDATE ... WHERE status = 'pending'` (atomic)
- Concurrent duplicate requests cannot both succeed

### Token Security
- Tokens written to DB, read once by web via Realtime
- After web consumes tokens, status transitions make them inaccessible
- Alternative: clear token fields after consumption

### QR Code Safety
- Contains only Mini Program path + UUID (e.g., `pages/login/index?uuid=xxx`)
- No sensitive data in QR content
- UUID is crypto-random, unguessable

### Request Validation
- Edge Function checks parameter completeness
- Same UUID only accepts one valid request

## Edge Function Interface

### `POST /functions/v1/wechat-login`

**Request:**
```json
{
  "code": "wx.login code from Mini Program",
  "uuid": "session UUID from web"
}
```

**Processing:**
1. Validate params (code, uuid non-empty)
2. Query wechat_login_sessions:
   - UUID not found → 400 "invalid session"
   - status ≠ pending → 400 "session already used"
   - expires_at < now() → 400 "session expired"
3. Call WeChat code2session API
4. errcode ≠ 0 → update status='failed', return 400
5. Get openid → lookup auth.users by user_metadata.wechat_openid
   - Not found → create new user (admin API)
   - Found → use existing user
6. Generate session for user
7. Update table: status='confirmed', user_id, access_token, refresh_token
8. Return 200

**Response:**
```json
{ "success": true }
// or
{ "success": false, "error": "session expired" }
```

## Mini Program (Minimal Implementation)

### Structure
Single page: `pages/login/index`

### Flow
1. Open from QR scan → get `uuid` from path params
2. Show "Confirm login to IceZone Studio" button
3. User taps → `wx.login()` → get code
4. POST {code, uuid} to Edge Function
5. Success → "Login successful, you may close"
6. Failure → error message + retry button

### UI States
- `idle`: Logo + confirm button
- `loading`: spinner
- `success`: checkmark + message
- `error`: error text + retry button

## Web Frontend UI

### Login Page
- Add "WeChat Scan Login" button below existing Google button

### QR Modal (on click)
- QR code (generated by `qrcode.react`)
- "Scan with WeChat" instruction text
- Countdown progress bar (5 min)
- States:
  - `pending`: QR + countdown
  - `confirmed`: modal closes, redirect to dashboard
  - `failed`: error message + "regenerate QR" button
  - `expired`: QR greyed out + "click to refresh" overlay

### QR Content Strategy
- Use "scan regular QR to open Mini Program" feature
- Configure URL rule in Mini Program backend
- QR contains URL like `https://<domain>/wechat-scan?uuid=xxx`
- WeChat recognizes the URL pattern → opens Mini Program with uuid param
